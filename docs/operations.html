Operations Runbooks

This page provides baseline operating procedures for StellaraeSecure services in self-hosted and production-like environments.

Scope

  • Services: staffdb, oauth2, 2fa, admin.
  • Runtime: containerized deployment using root compose files.
  • Data stores: SQLite volumes for each service.

Runbook: Backup

Create backups from quiesced services or use snapshot-capable volumes to avoid partial writes.

Scripted option: scripts/ops/backup.sh (supports volume overrides via env vars). 

# Stop write-heavy traffic first, then snapshot volumes.
docker compose --env-file .env.compose stop oauth2 2fa staffdb

# Example copy-out backup pattern (adjust paths for your host).
mkdir -p backups/$(date +%F)
docker run --rm -v stellaraesecure_staffdb-data:/from -v "$PWD"/backups/$(date +%F):/to alpine sh -c "cp -a /from /to/staffdb-data"
docker run --rm -v stellaraesecure_oauth2-data:/from -v "$PWD"/backups/$(date +%F):/to alpine sh -c "cp -a /from /to/oauth2-data"
docker run --rm -v stellaraesecure_twofa-data:/from -v "$PWD"/backups/$(date +%F):/to alpine sh -c "cp -a /from /to/twofa-data"

# Resume services.
docker compose --env-file .env.compose start staffdb oauth2 2fa

Runbook: Restore

  1. Stop all services and verify no writers are active.
  2. Restore volume contents from the target backup set.
  3. Start staffdb first, then 2fa, then oauth2/admin.
  4. Run health and readiness checks for all services.
  5. Verify a known account and token workflow before opening traffic.

Scripted option: scripts/ops/restore.sh <backup-directory>.

Runbook: Secret Rotation

Targets

  • STAFFDB_API_KEY
  • OAUTH2_ADMIN_API_KEY
  • TWOFA_API_KEY
  • OAUTH2_STAFF_IDENTITY_SIGNING_KEY
  • OAUTH2_TOKEN_SIGNING_KEY

Procedure

  1. Generate replacement values in a secure secret manager.
  2. Update `.env.compose` and any deployment secret stores.
  3. Roll services in dependency order: staffdb, 2fa, oauth2, admin.
  4. Run smoke checks for admin session issue/verify and OAuth token exchange.
  5. Revoke stale panel sessions if identity signing keys changed.

Runbook: Incident Response

Severity model

  • SEV-1: Auth bypass, key compromise, or broad token abuse.
  • SEV-2: Degraded authorization decisions or 2fa verification failures.
  • SEV-3: Single-node failures with no active security impact.

Immediate actions

  1. Capture affected correlation IDs and request samples.
  2. Contain by rotating exposed keys and revoking active panel sessions.
  3. Halt risky admin operations by restricting ingress to oauth2 admin paths.
  4. Restore known-good service versions if regression is confirmed.
  5. Publish incident timeline and remediation notes internally.

Monitoring and SLO Guidance

Metrics endpoints

curl http://127.0.0.1:3000/metrics
      curl http://127.0.0.1:4000/metrics
      curl http://127.0.0.1:4100/metrics

Recommended service-level indicators (SLIs)

  • Availability: percentage of successful `/ready` probes.
  • Latency: p95 response time for oauth2 `/api/token` and `/api/authorize`.
  • Error rate: 5xx ratio by service and endpoint family.
  • Policy health: rate of second-factor policy denials in oauth2 admin routes.

Initial SLO targets

SLO Target Window
Control-plane availability (oauth2 + staffdb + 2fa ready) 99.9% 30 days
oauth2 token issue latency p95 < 300ms 7 days
5xx error budget burn < 1% 30 days

Alerting baseline

  • Page on sustained readiness failures over 5 minutes.
  • Page on oauth2 5xx rates above 2% for 10 minutes.
  • Ticket on elevated second-factor denials over rolling baseline.
